## AWS Storage Types Overview

Understanding where S3 fits in the AWS storage landscape.

| Type | Services | Characteristics | Use Cases |
| :--- | :--- | :--- | :--- |
| **Block** | EBS, EC2 Instance Store | Fixed-size blocks, low latency | Databases, boot volumes, transaction workloads |
| **File** | EFS, FSx | Hierarchical, shared access | Concurrent access across instances |
| **Object** | [S3 (all classes including Glacier)](https://docs.aws.amazon.com/AmazonS3/latest/userguide/storage-class-intro.html) | Flat namespace, metadata-rich | Data lakes, backups, archives, static content |

Glacier is _not_ a separate storage type - it is part of S3 as archive storage classes.

## S3 Overview

Amazon S3 (Simple Storage Service) is object storage built to store and retrieve any amount of data from anywhere.

**Key use cases**:

- Cloud and hybrid cloud storage
- Backup and disaster recovery
- Archive and compliance
- Data lakes and big data analytics
- Static website hosting
- Media hosting and content distribution
- Software delivery

**S3 integration services**: [Amazon SageMaker](https://docs.aws.amazon.com/sagemaker/latest/dg/whatis.html) (ML training), [Amazon Athena](https://docs.aws.amazon.com/athena/latest/ug/what-is.html) (SQL queries on S3), [Amazon EMR](https://docs.aws.amazon.com/emr/latest/ManagementGuide/emr-what-is-emr.html) (big data processing), [AWS Glue](https://docs.aws.amazon.com/glue/latest/dg/what-is-glue.html) (ETL and data catalog).

## Bucket Types Comparison

S3 offers four bucket types optimized for different workloads.

| Bucket Type | Storage Structure | Availability | Storage Classes | Naming Scope | Primary Use Case |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **General Purpose** | Flat (prefix-based) | Multi-AZ | All except Express One Zone | Global (per partition) | Most workloads |
| **Directory** | Hierarchical | Single AZ | S3 Express One Zone | Regional with zone ID | Low-latency, performance-critical |
| **Table** | Apache Iceberg tables | Multi-AZ | Managed by S3 | Account + Region | Analytics, data lakes |
| **Vector** | Vector embeddings | Regional | Managed by S3 | Account + Region | AI/ML, semantic search, RAG |

### General Purpose Buckets

The original S3 bucket type supporting all storage classes except S3 Express One Zone. Objects are stored redundantly across [minimum 3 Availability Zones](https://docs.aws.amazon.com/AmazonS3/latest/userguide/DataDurability.html).

### Directory Buckets

Use [S3 Express One Zone storage class](https://docs.aws.amazon.com/AmazonS3/latest/userguide/directory-buckets-overview.html) with hierarchical namespace. [Up to 10x faster than S3 Standard](https://docs.aws.amazon.com/AmazonS3/latest/userguide/s3-express-performance.html) with significantly lower request costs ([55% reduction for PUT requests, 85% reduction for GET requests as of April 2025](https://aws.amazon.com/about-aws/whats-new/2025/04/amazon-s3-express-one-zone-reduces-storage-request-prices/)). Single-digit millisecond latency. [Naming format](https://docs.aws.amazon.com/AmazonS3/latest/userguide/directory-bucket-naming-rules.html): `bucket-base-name--zone-id--x-s3`.

### Table Buckets

[S3 Tables](https://docs.aws.amazon.com/AmazonS3/latest/userguide/s3-tables.html) store tabular data in [Apache Iceberg format](https://docs.aws.amazon.com/AmazonS3/latest/userguide/s3-tables.html) with automatic maintenance (compaction, snapshot management). Queryable via Athena, Redshift, and Apache Spark. Names are [unique within account and region only](https://docs.aws.amazon.com/AmazonS3/latest/userguide/s3-tables-buckets-naming.html) - _not_ globally unique.

### Vector Buckets

[S3 Vectors](https://docs.aws.amazon.com/AmazonS3/latest/userguide/s3-vectors.html) ([generally available as of December 2, 2025](https://aws.amazon.com/about-aws/whats-new/2025/12/amazon-s3-vectors-generally-available/)) store vector embeddings for AI agents, RAG, and semantic search. Supports [up to 2 billion vectors per index](https://docs.aws.amazon.com/AmazonS3/latest/userguide/s3-vectors-limitations.html) with subsecond query latency.

## Bucket Naming and Object Structure

### Bucket Naming Rules

[General purpose bucket naming requirements](https://docs.aws.amazon.com/AmazonS3/latest/userguide/bucketnamingrules.html):

| Rule | Specification |
| :--- | :--- |
| Length | [3-63 characters](https://docs.aws.amazon.com/AmazonS3/latest/userguide/bucketnamingrules.html) |
| Valid characters | Lowercase letters, numbers, periods (`.`), hyphens (`-`) |
| Start/End | Must begin and end with a letter or number |
| Prohibited | Two adjacent periods, IP address format (e.g., `192.168.5.4`) |
| [Reserved prefixes](https://docs.aws.amazon.com/AmazonS3/latest/userguide/bucketnamingrules.html) | `xn--`, `sthree-`, `amzn-s3-demo-` |
| [Reserved suffixes](https://docs.aws.amazon.com/AmazonS3/latest/userguide/bucketnamingrules.html) | `-s3alias`, `--ol-s3`, `.mrap`, `--x-s3`, `--table-s3` |

General purpose bucket names are globally unique within an AWS [partition](https://docs.aws.amazon.com/IAM/latest/UserGuide/reference-arns.html) (`aws`, `aws-cn`, `aws-us-gov`). S3 is _not_ the only globally unique resource in AWS - Route 53 domains and IAM account aliases also have global uniqueness requirements.

### Object Structure

S3 stores objects (files) in buckets (containers). [Objects](https://docs.aws.amazon.com/AmazonS3/latest/userguide/UsingObjects.html) consist of:

| Component | Description | Limits |
| :--- | :--- | :--- |
| **Key** | Full path (prefix + object name) | [Up to 1,024 bytes UTF-8](https://docs.aws.amazon.com/AmazonS3/latest/userguide/object-keys.html) |
| **Value** | Object content | Max [50 TB](https://aws.amazon.com/about-aws/whats-new/2025/12/amazon-s3-maximum-object-size-50-tb/) (increased from 5 TB on Dec 2, 2025) |
| **Metadata** | System and user-defined key-value pairs | [2 KB for user metadata](https://docs.aws.amazon.com/AmazonS3/latest/dev/UsingMetadata.html) |
| **Tags** | Key-value pairs for lifecycle and access | Up to [10 tags](https://docs.aws.amazon.com/AmazonS3/latest/userguide/object-tagging.html) per object |
| **Version ID** | System-assigned identifier | Only when versioning enabled |

[Directories are an abstraction](https://docs.aws.amazon.com/AmazonS3/latest/userguide/object-keys.html) - S3 uses a flat namespace where the key `photos/2024/image.jpg` is stored as a single key, not nested folders.

**Object URL format**: `s3://bucket-name/prefix/object.name`

### Multipart Upload

[Multipart upload](https://docs.aws.amazon.com/AmazonS3/latest/userguide/mpuoverview.html) is [recommended for objects 100 MB or larger](https://docs.aws.amazon.com/AmazonS3/latest/userguide/upload-objects.html) and required for objects over 5 GB (single PUT limit). Supports [up to 10,000 parts](https://docs.aws.amazon.com/AmazonS3/latest/userguide/qfacts.html).

### S3 Access Points

[S3 Access Points](https://docs.aws.amazon.com/AmazonS3/latest/userguide/access-points.html) are named network endpoints with dedicated access policies. Each access point has its own IAM resource policy that works in conjunction with the bucket policy. Use access points to simplify managing access for shared datasets at scale.

## Storage Classes

### Storage Classes Comparison

[S3 storage classes](https://docs.aws.amazon.com/AmazonS3/latest/userguide/storage-class-intro.html) provide different durability, availability, and cost characteristics:

| Storage Class | Durability | Availability | AZs | Min Duration | Retrieval Fee | Use Case |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| **S3 Standard** | 99.999999999% | 99.99% | ≥3 | None | None | Frequently accessed data |
| **S3 Intelligent-Tiering** | 99.999999999% | 99.9% | ≥3 | None | None | Unknown or changing access |
| **S3 Standard-IA** | 99.999999999% | 99.9% | ≥3 | 30 days | Per-GB | Infrequent but rapid access |
| **S3 One Zone-IA** | 99.999999999% | 99.5% | 1 | 30 days | Per-GB | Secondary backup copies |
| **S3 Express One Zone** | 99.999999999% | 99.95% | 1 | None | Per-GB | Performance-critical |
| **S3 Glacier Instant Retrieval** | 99.999999999% | 99.9% | ≥3 | 90 days | Per-GB | Archive with millisecond access |
| **S3 Glacier Flexible Retrieval** | 99.999999999% | 99.99%* | ≥3 | 90 days | Per-GB | Archive with hours retrieval |
| **S3 Glacier Deep Archive** | 99.999999999% | 99.99%* | ≥3 | 180 days | Per-GB | Long-term archive |

Note: Availability values marked with * apply after the restore operation completes.

See [S3 storage classes pricing page](https://aws.amazon.com/s3/pricing/) for current pricing details.

### Durability vs Availability

| Concept | Definition | Example |
| :--- | :--- | :--- |
| **Durability** | Probability data will NOT be lost over a year | 11 nines = expect to lose 0.000000001% of objects annually |
| **Availability** | Percentage of time service is accessible | 99.99% = ~52 minutes downtime per year allowed |

All [S3 storage classes provide 11 nines (99.999999999%) durability](https://docs.aws.amazon.com/AmazonS3/latest/userguide/storage-class-intro.html). Data is stored across a [minimum of 3 AZs](https://docs.aws.amazon.com/AmazonS3/latest/userguide/DataDurability.html) (except One Zone classes). AWS states S3 is designed to sustain data in the event of loss of an entire Availability Zone.

### Glacier Retrieval Options

[Glacier retrieval options](https://docs.aws.amazon.com/AmazonS3/latest/userguide/restoring-objects-retrieval-options.html):

| Tier | Glacier Flexible Retrieval | Glacier Deep Archive | Cost |
| :--- | :--- | :--- | :--- |
| **Expedited** | 1-5 minutes | Not available | Premium |
| **Standard** | 3-5 hours | 12 hours | Standard |
| **Bulk** | 5-12 hours | 48 hours | Free (Flexible) / Lowest (Deep) |

### S3 Intelligent-Tiering

[Intelligent-Tiering](https://docs.aws.amazon.com/AmazonS3/latest/userguide/intelligent-tiering-overview.html) automatically moves objects between access tiers based on usage patterns. Small monthly monitoring fee; no retrieval charges.

| Tier | Trigger | Retrieval |
| :--- | :--- | :--- |
| Frequent Access | Default / after access | Instant |
| Infrequent Access | 30 days no access | Instant |
| Archive Instant Access | 90 days no access | Instant (milliseconds) |
| Archive Access (optional) | 90-730 days configurable | Minutes to hours |
| Deep Archive Access (optional) | 180-730 days configurable | 12 hours |

[Objects smaller than 128 KB are not monitored](https://docs.aws.amazon.com/AmazonS3/latest/userguide/storage-class-intro.html) and always remain in the Frequent Access tier.

### Lifecycle Policies

[S3 Lifecycle](https://docs.aws.amazon.com/AmazonS3/latest/userguide/lifecycle-transition-general-considerations.html) rules automate transitions between storage classes. Transitions follow a waterfall pattern - you can move down tiers but not up. [S3 Express One Zone does not support lifecycle transitions](https://docs.aws.amazon.com/AmazonS3/latest/userguide/directory-buckets-objects-lifecycle.html).

## S3 Security

### Access Control Mechanisms

| Mechanism | Scope | When to Use |
| :--- | :--- | :--- |
| **IAM Policies** | User/role-based | Control what AWS principals can do |
| **Bucket Policies** | Resource-based | Cross-account access, public access, resource-level control |
| **ACLs** | Object/bucket level | Legacy - AWS recommends disabling |

[Access evaluation](https://docs.aws.amazon.com/AmazonS3/latest/userguide/access-control-auth-workflow-bucket-operation.html): Within a single account, a request is allowed if (an IAM identity-based policy allows OR a bucket policy allows) AND no explicit DENY exists.

For [cross-account access](https://docs.aws.amazon.com/IAM/latest/UserGuide/access_policies-cross-account-resource-access.html), you need both an allow in the requester's IAM identity-based policy AND an allow in the bucket's resource-based policy, and there must be no explicit DENY in any applicable policy.

AWS recommends using [Bucket owner enforced Object Ownership setting](https://docs.aws.amazon.com/AmazonS3/latest/userguide/about-object-ownership.html) ([default for new buckets](https://docs.aws.amazon.com/AmazonS3/latest/userguide/object-ownership-new-bucket.html)) which disables ACLs.

### Bucket Policy Structure

[Bucket policies](https://docs.aws.amazon.com/AmazonS3/latest/userguide/access-policy-language-overview.html) use JSON format:

```json
{
  "Version": "2012-10-17",
  "Statement": [{
    "Sid": "PublicReadGetObject",
    "Effect": "Allow",
    "Principal": "*",
    "Action": "s3:GetObject",
    "Resource": "arn:aws:s3:::bucket-name/*"
  }]
}
```

| Element | Description |
| :--- | :--- |
| **Version** | Policy language version (always `2012-10-17`) |
| **Statement** | Array of permission statements |
| **Effect** | `Allow` or `Deny` |
| **Principal** | Who receives permission (`*` for public, account ARN, etc.) |
| **Action** | S3 operations (e.g., `s3:GetObject`, `s3:PutObject`) |
| **Resource** | Bucket ARN or object ARN pattern |
| **Condition** | Optional restrictions (IP, VPC, encryption, etc.) |

### Block Public Access

[Block Public Access](https://docs.aws.amazon.com/AmazonS3/latest/userguide/access-control-block-public-access.html) settings provide an account or bucket-level safety net. Can be applied at organization, account, bucket, or access point level.

| Setting | API Name | Effect |
| :--- | :--- | :--- |
| Block new public ACLs | `BlockPublicAcls` | Prevents creating new public ACLs |
| Ignore all public ACLs | `IgnorePublicAcls` | Treats all ACLs as non-public |
| Block new public policies | `BlockPublicPolicy` | Rejects bucket policies allowing public access |
| Block public and cross-account | `RestrictPublicBuckets` | When bucket has a public policy, restricts access to AWS service principals and authorized users within the bucket owner account; blocks all cross-account access including explicitly granted permissions |

**Important**: Disabling Block Public Access does _not_ automatically make a bucket public - you must still explicitly create a policy granting public access. "CAN be public" does not equal "IS public."

### Encryption Options

| Type | Key Management | Default? |
| :--- | :--- | :--- |
| **SSE-S3** | S3-managed keys | Yes (since [Jan 5, 2023](https://docs.aws.amazon.com/AmazonS3/latest/userguide/default-encryption-faq.html)) |
| **SSE-KMS** | AWS KMS keys | No - customer enables |
| **DSSE-KMS** | Dual-layer KMS encryption | No - customer enables |
| **SSE-C** | Customer-provided keys | No - [disabled by default starting April 6, 2026](https://aws.amazon.com/blogs/storage/advanced-notice-amazon-s3-to-disable-the-use-of-sse-c-encryption-by-default-for-all-new-buckets-and-select-existing-buckets-in-april-2026/) |
| **Client-side** | Customer manages externally | No - customer implements |

All objects uploaded to S3 are [automatically encrypted with SSE-S3](https://docs.aws.amazon.com/AmazonS3/latest/userguide/default-encryption-faq.html) unless you specify a different encryption method.

### Pre-signed URLs

[Pre-signed URLs](https://docs.aws.amazon.com/AmazonS3/latest/userguide/ShareObjectPreSignedURL.html) grant temporary access to specific objects using the credentials of the URL generator.

| Feature | Pre-signed URL | Public Access |
| :--- | :--- | :--- |
| Duration | [1 second to 7 days (CLI), 12 hours (console)](https://docs.aws.amazon.com/AmazonS3/latest/userguide/ShareObjectPreSignedURL.html) | Permanent until policy removed |
| Scope | Specific object(s) | All objects matching policy |
| Authentication | Uses generator's credentials | None required |
| Use case | Temporary download/upload links | Static websites, public assets |

### IAM Access Analyzer for S3

[IAM Access Analyzer](https://docs.aws.amazon.com/AmazonS3/latest/userguide/access-analyzer.html) identifies S3 buckets with external access (public or cross-account). Evaluates bucket policies, ACLs, access point policies, and multi-region access point policies. [External access analyzer findings update within 30 minutes of policy changes](https://docs.aws.amazon.com/IAM/latest/UserGuide/access-analyzer-findings-remediate.html).

## Versioning

[S3 Versioning](https://docs.aws.amazon.com/AmazonS3/latest/userguide/versioning-workflows.html) preserves, retrieves, and restores every version of every object in a bucket.

**Bucket states**: Unversioned (default), Enabled, Suspended

| Scenario | Behavior |
| :--- | :--- |
| [Upload before versioning enabled](https://docs.aws.amazon.com/AmazonS3/latest/userguide/versioning-workflows.html) | Object has version ID `null` |
| DELETE without version ID | Creates a delete marker (hides object) |
| DELETE with specific version ID | Permanently deletes that version |
| GET on delete-marked object | Returns 404 with `x-amz-delete-marker: true` header |
| Suspend versioning | New uploads get version ID `null`; existing versions remain |

**Delete markers**: When you delete an object without specifying a version ID, S3 inserts a delete marker as the current version. The object appears deleted but previous versions remain. To restore, delete the delete marker by specifying its version ID.

Suspending versioning does _not_ delete existing versions - they remain accessible by version ID.

## Replication

[S3 Replication](https://docs.aws.amazon.com/AmazonS3/latest/userguide/replication.html) automatically copies objects across buckets.

| Type | Buckets Location | Data Transfer Charges | Use Cases |
| :--- | :--- | :--- | :--- |
| **CRR** (Cross-Region) | Different AWS Regions | Yes (inter-region data transfer, replication requests, storage) | Compliance, latency reduction, disaster recovery |
| **SRR** (Same-Region) | Same AWS Region | No inter-region data transfer (replication requests and storage charges still apply) | Log aggregation, prod/test sync, data sovereignty |

**[Requirements](https://docs.aws.amazon.com/AmazonS3/latest/userguide/replication.html)**:

- Versioning must be enabled on both source and destination buckets
- S3 must have IAM permissions to replicate objects
- Source bucket owner needs read access to objects being replicated

**Cross-account replication**: Supported - destination bucket owner must grant source account permissions via bucket policy.

**Note**: [Live replication does not replicate existing objects](https://docs.aws.amazon.com/AmazonS3/latest/userguide/s3-batch-replication-batch.html). Use [S3 Batch Replication](https://docs.aws.amazon.com/AmazonS3/latest/userguide/s3-batch-replication-batch.html) for pre-existing objects.

## Static Website Hosting

S3 can host [static websites](https://docs.aws.amazon.com/AmazonS3/latest/userguide/HostingWebsiteOnS3Setup.html) with HTML, CSS, JavaScript, and media files.

**Configuration steps**:

1. Create an S3 bucket
2. Enable static website hosting
3. Disable Block Public Access settings
4. Add bucket policy granting `s3:GetObject` to `Principal: "*"`
5. Configure index document (e.g., `index.html`)
6. Optionally configure error document (e.g., `404.html`)

**Website endpoint format**: Varies by region - either `bucket-name.s3-website-Region.amazonaws.com` (dash format, e.g., us-west-2) or `bucket-name.s3-website.Region.amazonaws.com` (dot format, e.g., ap-south-1). See [website endpoints documentation](https://docs.aws.amazon.com/AmazonS3/latest/userguide/WebsiteEndpoints.html) for region-specific formats.

**Common 403 error causes**:

- Block Public Access still enabled
- Missing or incorrect bucket policy
- Incorrect resource ARN in policy (should be `arn:aws:s3:::bucket-name/*`)

**HTTPS and CloudFront patterns**:

- [S3 website endpoints are HTTP-only](https://docs.aws.amazon.com/AmazonS3/latest/userguide/WebsiteEndpoints.html) and [cannot be used with CloudFront Origin Access Control (OAC)](https://docs.aws.amazon.com/AmazonCloudFront/latest/DeveloperGuide/private-content-restricting-access-to-s3.html)
- For HTTPS with a private bucket, use CloudFront in front of the S3 REST endpoint (e.g., `bucket-name.s3.region.amazonaws.com`) with OAC, and configure default root object and custom error responses at CloudFront for index and error routing
- If using the S3 website endpoint directly for its built-in index and error document features, your bucket must be publicly accessible and content will only be available over HTTP
- Recommended: Use [AWS Amplify Hosting](https://docs.aws.amazon.com/amplify/latest/userguide/welcome.html) for fully managed static website hosting with HTTPS via CloudFront

## Data Migration and Hybrid Storage

### AWS Snow Family

Physical devices for offline data transfer and edge computing.

| Device | Storage | Compute | Use Case |
| :--- | :--- | :--- | :--- |
| **[Snowcone](https://docs.aws.amazon.com/snowball/latest/snowcone-guide/snowcone-spec-requirements.html)** | 8 TB HDD / 14 TB SSD | 2 vCPUs, 4 GB RAM | Portable edge computing, small migrations |
| **[Snowball Edge Storage Optimized](https://docs.aws.amazon.com/snowball/latest/developer-guide/device-differences.html)** | Up to 210 TB | Limited | Large-scale data migration |
| **[Snowball Edge Compute Optimized](https://docs.aws.amazon.com/snowball/latest/developer-guide/device-differences.html)** | Up to 104 vCPUs, 416 GB RAM | High | Edge computing, ML at edge |

**Important**: Effective [November 7, 2025](https://docs.aws.amazon.com/snowball/latest/developer-guide/snowball-edge-availability-change.html), [Snowball Edge devices](https://aws.amazon.com/snowball/) are only available to existing customers. New customers should use AWS DataSync for online transfers or AWS Data Transfer Terminal for physical transfers.

**Pricing**: Pay for device usage and data transfer out. [Data import to S3 is free](https://aws.amazon.com/snowball/pricing/).

### AWS Storage Gateway

[Storage Gateway](https://docs.aws.amazon.com/storagegateway/latest/vgw/WhatIsStorageGateway.html) bridges on-premises storage with AWS cloud storage.

| Gateway Type | Protocol | Backend | Use Case |
| :--- | :--- | :--- | :--- |
| **S3 File Gateway** | NFS, SMB | Amazon S3 | File access to S3 with local caching |
| **Volume Gateway (Cached)** | iSCSI | S3 (primary), local cache | Hot data local, cold in cloud |
| **Volume Gateway (Stored)** | iSCSI | Local (primary), S3 snapshots | All data local, cloud DR |
| **Tape Gateway** | VTL (iSCSI) | S3 Glacier | Replace physical tape backup |

FSx File Gateway was [discontinued for new customers on October 28, 2024](https://docs.aws.amazon.com/filegateway/latest/filefsxw/DocumentHistory.html).

## Shared Responsibility Model

### AWS Responsibilities

- Infrastructure management across global data centers
- Sustaining data through loss of an entire Availability Zone
- Configuration and vulnerability analysis of infrastructure
- SLA compliance validation

### Customer Responsibilities

- Enabling and configuring versioning
- Creating and managing bucket policies and IAM policies
- Setting up replication rules
- Configuring logging and monitoring
- Choosing appropriate storage classes and lifecycle rules
- Encrypting data at rest (beyond default SSE-S3) and in transit

## Best Practices

- Enable versioning on production buckets to protect against accidental deletion
- Use S3 Intelligent-Tiering for data with unknown or changing access patterns
- Configure lifecycle policies to automatically transition infrequently accessed data to lower-cost storage classes
- Enable Block Public Access at the account level unless you specifically need public buckets
- Use bucket policies over ACLs - ACLs are a legacy access control mechanism
- Enable server-side encryption (enabled by default with SSE-S3)
- Use S3 Access Points to simplify access management for shared datasets
- Configure S3 replication for disaster recovery and compliance requirements
- Use pre-signed URLs for temporary access rather than making objects public
- Monitor bucket access with IAM Access Analyzer to identify unintended external access
